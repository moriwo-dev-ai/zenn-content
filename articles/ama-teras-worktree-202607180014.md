## 導入: 自己進化という言葉の誤解

AMA-terasでは「自己進化」という言葉を使っています。

ただし、これは「AIが勝手に本体を書き換える」という意味ではありません。  
足りない機能に出会ったとき、実装案を生成し、隔離した `git worktree` で検証し、typecheck・テスト・実起動スモークを通し、最後に人間が承認してから本体へ昇格する、という意味です。

外部への発信や書き込みも同じです。  
Zennへの公開、GitHub Releaseの公開など、外に出る操作は承認ダイアログを通します。下書きの作成までは自動化できても、公開ボタンを押すのは人間です。

自己進化の中心にあるのは「自動で変わること」ではなく、「安全に候補を作り、安全に検証し、安全に昇格できること」です。

## AMA-terasの自己進化フロー

### 足りない機能に出会う

AMA-terasは、対話しながら計画、実装、デバッグ、テストを進めるElectron製のデスクトップAIコーディングエージェントです。

作業中に「いまの道具では足りない」場面に出会うことがあります。  
そのとき、いきなり本体を変更するのではなく、まず追加すべき機能や修正を候補として扱います。

### 自分で実装を生成する

候補に対して、AMA-terasは実装を生成します。

ここで生成されるものは、まだ本体ではありません。  
本体に混ざる前の「候補」です。

この段階で重要なのは、生成したという事実と、採用するという判断を分けることです。  
生成できたから採用するのではなく、検証を通り、人間が承認して初めて昇格できます。

### 隔離したgit worktreeで検証する

生成物は隔離した `git worktree` に置かれます。

本体の作業ツリーに直接混ぜないことで、候補の実装が壊れていても本体を壊しません。  
失敗した候補は、候補として失敗します。本体の状態とは分離されます。

### typecheck・テスト・実起動スモークを通す

候補は、少なくとも次の検証ゲートを通ります。

- typecheck
- テスト
- 実起動スモーク

typecheckだけでは足りません。  
テストだけでも足りません。  
実際に起動して動くかも確認します。

PROGRESS.mdには、テストでは捕まらなかった実アプリ経路のバグも記録されています。

M72では、Blueskyの検索だけが401になる問題がありました。原因はapp passwordではなく、`BlueskyReader` の既定 `fetchImpl` が第2引数の `Authorization` ヘッダを捨てていたことでした。テストはすべて `fetch` を注入していたため、本番だけが既定値を使い、検索だけが無認証で飛んでいました。

この修正では、bluesky / zenn / hn / hatena の既定を直し、注入なし経路、つまり実アプリの経路をテストで固定しています。さらに実機で巡回成功と検索10件を確認しています。

「テストが通った」だけではなく、「実アプリの経路が通った」ことを確認する必要がある、という記録です。

### 本体への昇格は人間が承認する

検証を通った候補も、自動で本体に昇格するわけではありません。

本体への昇格は人間の承認を経ます。  
AMA-terasの自己進化は、生成、検証、承認を分けています。

この分離があるため、「自分で道具を作る」ことと「勝手に本体を書き換える」ことは同じではありません。

## 直近の実例

この記事では、提供されたPROGRESS.md抜粋で確認できる範囲だけを扱います。  
job-38 / job-42 / job-47、M94、coreプロンプトへの還元については、この抜粋内に検証できる記録がないため本文では扱いません。

### M73: Zenn記事の公開をアプリから行えるようにした

M73では、Zenn記事の「記事化」が `published: false` でコミットするところまでで止まっていたことが記録されています。

GitHub Releaseはスマホから公開できる一方で、Zenn記事はコミット後の公開に手が届いていませんでした。  
そこで `zenn-repo` に `publish-article` が追加されました。

この操作は、岩戸ゲートで全文を確認してから実行されます。  
承認後に `published: true` にしてpushします。

ここでも、外部へ出る操作は自動実行ではなく承認を通します。

### M74: スマホ承認経路で重複コミットが起きた

M74では、スマホからの承認経路で重複が起きたことが記録されています。

実害として、23:05:05、23:05:07、23:05:10、23:06:26の流れで、5秒間に3本Zennへコミットされ、同じ記事が2本公開リポジトリに載りました。

原因は、M69で入れた双子まとめが `batchRespond` にしかなく、スマホは1枚でも `bulkRespond` を通っていたことです。  
さらに、下書きの「投稿済み」印は実行完了後に付くため、本文生成の数秒間に重複ガードを競合ですり抜けました。

修正として、`bulkRespond` の直列化と双子まとめが実装されました。

これは、承認ダイアログがあっても、承認経路が複数ある場合は同じガードを通さなければならない、という失敗です。

### M76: 「出したつもり」と「読める」は別だった

M76では、3本を `published: true` にしてpushしたものの、Zennが同期したのは2本だけだったことが記録されています。

残り1本はZenn側で403、つまり存在するが非公開で、誰も読めない状態でした。

しかし台帳は、実際に公開された2本を「公開待ち」のまま残し、Zennが同期していない1本を「投稿済み」と記録していました。

修正として、`ZennReader.isLive(slug)` が追加されました。

- 200: 読める
- 403 / 404: 読めない
- 不達: 不明

台帳は、実際に読めるものだけを `posted` にするようになりました。  
公開ボタンも、pushしただけで「公開した」とは言わないようになりました。

### M77・M81・M83: 再デプロイも安全側に寄せた

M77では、Zennの投稿数上限により、記事が `published: true` なのに誰も読めない状態で固まったことが記録されています。

この状態では、アプリ上では公開待ちに出ず、公開ボタンも出ないのに、世には出ていませんでした。  
そこで `redeploy-article`、`zennStuck()`、PC/スマホの「再デプロイ」ボタンが追加されました。

ただし、M81で空コミットではZennが何も再デプロイしないことが分かりました。  
Zennは変更されたファイルを見て記事を同期するため、空コミットでは同期対象が存在しませんでした。  
修正として、記事ファイル自体に末尾の改行1文字だけ差分を付けるようにしました。本文は変えていません。

さらにM83では、再デプロイを押すほど遅くなる問題が記録されています。  
Zennは直近24時間に5本以上の投稿をブロックします。再デプロイもデプロイなので、詰まっているときに押すほど窓に積み上がり、解けなくなります。

実機では24時間に10本、上限の倍が積まれていました。

修正として、押す前に `git log` で窓を数え、上限に届いていれば実行しないようにしました。  
また、いつ枠が空くかを表示するようにしました。

### M85〜M87: リリース公開ボタンが消えていた

v1.2.0の下書きリリースはできていたものの、アプリ画面に公開ボタンが出てこない問題がありました。

M85-1では、`releaseInfo` の `useEffect` の依存が `[repo]` だけだったため、アプリを開いたまま下書きリリースを作ると、UIが起動時の `pendingDraft: null` を持ち続けることが分かりました。  
修正として、PC・スマホともに30秒ごとに取り直すようになりました。

M85-2では、IPCの許可リストが `draft / posted / discarded` しか知らず、M57で追加された `staged` をエラーにもせず捨てていたことが分かりました。  
修正として、知らない値は黙って捨てず、はっきり弾くようになりました。

M86では、公開ボタンが「公開できる状態になった瞬間に消える」ことが分かりました。  
「下書きリリースを公開」ボタンは下書きカードの操作行にあり、その操作行は `status === 'draft'` のときだけ描画されていました。  
しかしリリースにした下書きは `staged` になります。  
そのため、公開すべき状態になった瞬間にボタンが消えていました。

修正として、下書きカードから切り離し、独立セクションにしました。

M87では、公開したリリースを台帳が「公開待ち」と言い続ける問題が記録されています。  
そもそも、どの下書きがどのリリースなのか突き合わせる鍵がありませんでした。  
修正として `OperationsDraft.tag` を追加し、リリース作成時にタグを覚えるようにしました。  
また、GitHubの一次情報である `draft=false` を見て台帳を書き直すようにしました。
