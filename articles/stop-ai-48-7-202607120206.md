## TL;DR

- `GodScheduler.stop()` が `clearInterval` しかしておらず、起動遅延用の `setTimeout` を止め忘れていた
- 設定保存のたびに `operations.reset()` → 再init が走り、停止したはずの旧インスタンスが90秒後に蘇って増殖した
- 2026-07-12 00:00〜00:01 の48秒間に、AI会議(神議)が7回連続実行(各2.3〜3.1kトークン・計約19k)、承認バッチが6枚生成、手で直したはずの `schedule.json` が古い状態に巻き戻った
- 修正の核心は「タイマーを止める」ことではなく「**停止済みインスタンスは書き込まない**」を構造で保証したこと

## AMA-terasの自律運営とスケジューラ

AMA-terasには「神々の時計(AMENO-koyane)」と呼ぶスケジューラがあり、観測・巡回・下書き生成・神議(AI会議)といった運営タスクを定期実行している。神議はLLMを使って能力ギャップや発信の提案を出し、承認バッチとして人間に渡す。**外部への発信はすべて岩戸ゲート(承認ダイアログ)を通る**ので、幽霊が何体いようと外に何かが出ることはない——それでもLLMは回るので、トークンは実費で燃える。

### なぜ起動90秒後に setInterval を張る設計だったか

`GodScheduler.start()` は、起動直後の負荷を避けるため「起動90秒後に `setInterval` を張る」という構造になっていた。つまりこうだ。

```typescript
start() {
  setTimeout(() => {
    this.timer = setInterval(() => this.tick(), INTERVAL_MS);
  }, 90_000); // 起動90秒後から定期実行
}

stop() {
  clearInterval(this.timer); // ← setTimeout は誰も止めていない
}
```

`stop()` は `clearInterval` しかしていなかった。**setInterval を張る前の90秒間に stop() が呼ばれると、止めるべきタイマーはまだ存在せず、setTimeout だけが生き残る。** 90秒後、そのコールバックが発火して setInterval を張り、「停止したはずのスケジューラ」が蘇る。

## 事件: 2026-07-12 00:00の48秒間

### 症状1: 神議7回連続実行と承認バッチ6枚

2026-07-12 00:00〜00:01 の48秒間に、神議が7回連続で実行された。各2.3〜3.1kトークン、計約19kトークン。承認バッチが6枚生成された。

神議は本来、スケジュールに従って間隔を空けて走るタスクだ。48秒に7回という密度は、**同じ schedule.json を見ている複数のスケジューラインスタンスが、それぞれ独立に「実行時刻が来た」と判断して走った**ことを意味する。

### 症状2: 手で直したはずの schedule.json が巻き戻る

もう一つの症状が不気味だった。M38 で直接編集した `schedule.json`(神議に停止されていた omoi-kami を enabled=true で再開した変更)が、**古い状態に巻き戻っていた**。

これは幽霊スケジューラがそれぞれ古いメモリ状態を持ったままファイルを上書きしていたためだ。編集後にファイルを読んだインスタンスは正しい状態を持つが、編集前に読んだ幽霊が後から `save()` すると、ディスク上の正しい状態が古いメモリ内容で潰される。

### 実データで発覚した——テストでは捕まらなかった理由

このバグはテストではなく実データで発覚した。再現には「start() → 90秒経過前に stop() → 90秒経過」という時間依存のシーケンスが必要で、しかも増殖の引き金は「設定保存」という一見無関係な操作だった。当時のテストは stop() 後に setInterval が張られないことを検証しておらず、幽霊は本番でだけ生きていた。

## 原因: setTimeout の中の setInterval

### 設定保存のたびに幽霊が1体増える

増殖のメカニズムはこうだ。

1. `secretsSet` 等で設定を保存すると `operations.reset()` が呼ばれる
2. reset は既存スケジューラの `stop()` を呼ぶが、setTimeout は止まらない
3. 次のアクセスで再init され、新しいスケジューラが start() する
4. 90秒後、**旧インスタンスの setTimeout が発火して setInterval を張る**
5. これで生きているスケジューラが2体になる。設定保存のたびに1体ずつ増える

### 全幽霊が同じ schedule.json を古いメモリ状態で上書き

各幽霊は同じ `schedule.json` を見て tick を回すので、実行時刻が来ればそれぞれ神議を走らせる(→ 48秒で7回)。そして各自のメモリ状態でファイルを上書きするので、後から発火した幽霊が古い状態を書き戻す(→ 巻き戻り)。

トークンを浪費するだけでなく、**人間が手で施した運営状態の修正を静かに消していた**のがこのバグの実害だ。

## 修正: 止めるのではなく「書けなくする」

修正は2段構えにした。

### 1. startTimer を保持して clearTimeout / stopped フラグで復活を禁止

`setTimeout` の戻り値を `startTimer` として保持し、`stop()` で `clearTimeout` する。さらに `stopped` フラグを立て、一度止めたインスタンスは二度と start できない**使い捨て設計**にした。

```typescript
stop() {
  this.stopped = true;
  clearTimeout(this.startTimer); // 90秒待ちの setTimeout も止める
  clearInterval(this.timer);
}
```

### 2. tick() と save() を stopped で早期return——巻き戻しの構造的防止

こちらが修正の核心。タイマーを漏れなく止めるのは「頑張り」であって保証ではない。そこで **tick() と save() の先頭で stopped なら早期return** することにした。

```typescript
private tick() {
  if (this.stopped) return; // 停止済みは実行しない
  // ...
}

private save() {
  if (this.stopped) return; // 停止済みは書き込まない
  // ...
}
```

これで、仮に将来また止め忘れたタイマーが生まれても、**停止済みインスタンスがファイルを上書きすることは構造的に不可能**になる。「止める」の網羅性に頼らず、「書けない」で防ぐ。

### 偽タイマーによる再現テスト4件

偽タイマー(fake timers)で「start → 90秒経過前に stop → 時間を進める」というシーケンスを再現し、幽霊が蘇らないこと・停止済みインスタンスが書き込まないことを4テストで固定した。

## 教訓

### ライフサイクルを持つオブジェクトは「復活しない」ことをテストする

start/stop を持つオブジェクトのテストは「動くこと」「止まること」で満足しがちだが、本当に危ないのは「**止めた後に蘇らないこと**」だ。特に遅延起動(setTimeout の中で setInterval)のような多段タイマーは、stop() が全段を止めているか疑うべきだ。

### 「停止済みインスタンスからの書き込み禁止」は二重の防壁になる

タイマーの clear は漏れうる。しかし「stopped なら書かない」は、タイマーがどう漏れても最後の一線で守る。**副作用の入口(tick / save)で状態を確認する早期return**は、ライフサイクル管理のバグに対する安価で確実な防壁になる。

### 実コスト(約19kトークン)が出たからこそ、失敗もPROGRESSに残す

このバグは約19kトークンの実費と、手動修正の巻き戻しという実害を出した。だからこそ症状・原因・修正を PROGRESS.md にそのまま残している。外部発信は岩戸ゲートが守っていたので事故は内側で完結したが、「ガードが機能した」ことと「バグがなかった」ことは別の話だ。

## おまけ: 同日に直したもう一つの「別々の壊れ方」

同じ M41 で、投稿リンクの開き方がデスクトップとスマホで**それぞれ別の理由で壊れていた**のも直した。

### デスクトップ: window.open がアプリ内ウィンドウを開いていた

`window.open` が Electron のアプリ内ウィンドウを開いており、そこには X のログインセッションが無いので使えなかった。`openExternal` IPC(http/https のみ許可)で**既定ブラウザ**に開くよう変更し、さらに `setWindowOpenHandler` でアプリ内ウィンドウの生成自体を禁止した(防御)。

### スマホ: 承認後の window.open はポップアップブロックで必ず失敗

スマホ(リモート運営)では、承認という**通信のあと**に `window.open` していたため、ユーザージェスチャの文脈が切れてポップアップブロックに必ず引っかかっていた。承認後に「タップで1枚ずつ開くボタン」を並べる方式に変更した。タップ=ジェスチャなのでブロックされない。

同じ「リンクが開かない」でも、原因はプラットフォームごとにまったく別物だった。症状が同じでも原因を個別に追う——幽霊スケジューラと合わせて、この日の教訓はそれに尽きる。
