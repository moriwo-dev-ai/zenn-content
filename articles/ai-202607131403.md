## 動いていないのに「動いた」と報告していた話

AMA-teras は、対話しながら計画→実装→デバッグ→テストを進め、足りない機能に出会うと自分でツールを生成して取り込む(自己進化する)デスクトップAIコーディングエージェントです。この記事は、その AMA-teras が**動いていないのに「動いた」と報告していた**実話と、それをどう直したかの記録です。

### 403を握りつぶしていたコード

問題が起きたのは外部への投稿処理でした。投稿APIが 403 を返しているのに、コードはそのエラーを握りつぶし、処理全体としては成功扱いになっていました。結果、報告は「投稿しました」。実際には何も出ていません。

模式化すると、こういう構造でした。

```typescript
// 修正前(模式化):失敗しても成功として報告される
try {
  await postToExternal(payload);
} catch {
  // ここで握りつぶしていた
}
return { ok: true, message: "投稿しました" };
```

エージェント本人に悪意はありません。「投稿処理を呼び出した」ことと「投稿が成立した」ことを区別しないコードを書いてしまえば、報告は自動的に嘘になります。

### 気づいたきっかけと実機での確認

きっかけは、報告と実物の突き合わせでした。「投稿しました」という報告に対して投稿先を実機で確認したところ、投稿は存在しない。ログを遡ると 403 が返っていました。

「テストが通った」「エラーが出なかった」ではなく、**実機で結果物そのものを確認する**。このプロジェクトでは実機スモークを検証の一段として必ず置いていますが、この件は「実機で確認しないと嘘に気づけない」ことを自分たちに叩き込んだ事例になりました。

### 直し方:「出した」と「出す準備ができた」を分ける

修正の本質は、状態の語彙を分けることでした。

- **「出した」**: APIが成功レスポンスを返した事実がある場合だけ
- **「出す準備ができた」**: ペイロードの生成・検証までは完了したが、送信は成立していない場合
- それ以外はエラーとしてそのまま表に出す(握りつぶさない)

「成功」という一語に両方を押し込んでいたのが元凶です。準備完了を成功と報告する経路を潰し、エラーレスポンスは呼び出し元まで必ず届くようにしました。

## 自己進化はworktreeの中でしか起きない

「AIが自分でツールを生成して取り込む」と聞くと不安になると思います。上の403の件のような嘘が本体に混ざったら終わりだからです。だから AMA-teras の自己進化は、構造的に閉じ込めてあります。

### 検証ゲート(typecheck・テスト・実機スモーク)を通らないと昇格しない

生成物は本体のツリーではなく、**隔離された git worktree** の中に作られます。そして本体への昇格候補になるには、

1. typecheck
2. テスト
3. 実起動スモ
