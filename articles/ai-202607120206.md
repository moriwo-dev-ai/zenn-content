## はじめに: Star 0のプロジェクトがなぜ安全設計の話をするのか

まず現状の数字を正直に書きます。

- GitHub Star: **0**
- registry の clone: **10**
- Zenn のいいね: **2**
- 公開プラグイン: **1件**

誰も見ていないに等しい規模のプロジェクトが、なぜわざわざ安全設計の記事を書くのか。理由は単純で、**数字が小さい今のうちに、信頼の土台になる「ガードの実話」を残しておきたい**からです。

AMA-teras は Electron 製のデスクトップAIコーディングエージェントです。対話しながら計画→実装→デバッグ→テストを進め、足りない機能に出会うと自分でツールを生成して取り込む(自己進化する)設計を持っています。生成物は隔離した git worktree で typecheck・テスト・実起動スモークの検証ゲートを通り、本体への昇格は必ず人間の承認を経ます。

自己進化するエージェントに「プロジェクト運営」まで任せるとなると、問われるのは機能ではなくガードです。この記事は、そのガード=岩戸ゲートを「承認フローがあります」以上のものにするために何をやったか、そしてやらかしかけた話の記録です。

## 問題設定: AIに運営を任せる時、何が怖いか

最悪シナリオは具体的です。

- エージェントが**勝手にSNSへ投稿する**
- エージェントが**勝手にPRをマージする**

こうした事故に対して「承認フローを作りました」と言うだけでは弱い。承認UIの手前にある関数を直接呼べるなら、承認フローはただの飾りです。LLMが生成したコードや、将来の自分(自己進化で追加されるツール)がその関数に到達できる限り、バイパス経路は存在し続けます。

必要なのは「承認を通ってください」というお願いではなく、**承認なしで実行に到達する経路がコード上に存在しない**ことでした。

## 岩戸ゲート: 承認なしで実行に到達する経路をコードから消す

外部への発信・書き込みはすべて岩戸ゲート(承認ダイアログ)を通ります。その実装で重視した3点です。

### executorをWeakMapに封印する

実行関数(executor)は登録時に岩戸ゲート内部の WeakMap に封印し、**呼び出し元が持っていた参照からも削除**します。

```typescript
// 概念図: 登録後、ゲートの外に executor への参照が残らない
register(adapter: Adapter) {
  executorVault.set(adapter, adapter.execute); // WeakMapに封印
  delete adapter.execute;                       // 呼び出し元からも消す
}
```

以降、executor に触れるのはゲート内部のみで、そこには「プレビュー表示 → 人間の承認」が必ず挟まります。プレビューと承認を通らずに実行する経路が、そもそも参照として存在しません。

一括承認(M39)を足したときも、この掟は緩めていません。承認ダイアログは1回ですが**全件の全文プレビューを並べ**、未承認なら executor は1件も動きません。実行と audit は1件ずつで、1件失敗しても残りを続行し「N件中M件成功」を返します(部分成功を成功に見せない)。

### 宣言と実装の一致を登録時に検査

アダプタが宣言する能力と、実際に登録される executor の一致を登録時に検査します。

ここで象徴的なのが Bluesky の扱いです。「投稿できます」と宣言しておいて実行時に未実装エラーを返す、という形は採りませんでした。**宣言ごと空にする**。宣言と実挙動がズレた状態をコードベースに置かない、対応するときは executor と宣言を同時に足す、という方針です。

同じ理由で、X / はてブは岩戸ゲートを通りません。アダプタの execute 宣言が空(規約)なので、アプリは何も発行せず、URLを返して投稿画面を開くだけ。**最後のクリックは人間**です。

### 拒否も含めて全部記録(audit.jsonl)

承認された実行だけでなく、**拒否も含めてすべて** audit.jsonl に追記します。

```jsonl
{"ts":"...","action":"x:open-intent","decision":"approved","preview":"..."}
{"ts":"...","action":"github:release-draft","decision":"denied","preview":"..."}
```

また、承認要求には**10分のタイムアウト=拒否扱い**を設けています。人間が席を外している間に実行が宙吊りのまま溜まり続ける、という状態を作らないためです。無人放置は「保留」ではなく「拒否」として記録に落ちます。

## 神議アーキテクチャ: 自律の範囲を2段階制で線引きする

運営側では複数の役割(「神」と呼んでいます。観測・巡回・下書き生成・トリアージなど)がスケジュール実行され、定期的に「神議」で運営方針を提案します。ここで**自律でやってよいこと**と**承認が要ること**を2段階で線引きしました。

### 自律でできること・承認が要ること

| 自律でできる | 人間の承認が要る |
|---|---|
| 実行間隔の調整 | 新ツールの追加 |
| 監視キーワードの変更 | 判定プロンプトの変更 |
| 自分の停止 | 予算の引き上げ |
| 予算の**引き下げ** | 神の新設・削除 |

方向性がはっきりしています。**縮む方向・止まる方向は自律、広がる方向・強くなる方向は承認**です。

### 「騙り増額」対策: 宣言でなく実値で再分類

抜け道として想定したのが「予算引き下げです」と宣言して実際には引き上げる、いわば騙り増額です。対策は宣言を信用しないこと。**提案の宣言ラベルではなく、変更前後の実値を比較して引き下げ/引き上げを再分類**します。実値で引き上げなら、宣言が何であれ承認側に回ります。

### 神の新設は二重承認+テストで固定

神の新設はもっとも影響が大きい操作なので二重承認にしました。さらに「承認を拒否したら状態が書き換わらないこと」**それ自体をテストとして固定**しています。ガードは実装した瞬間より、後の変更で静かに壊れるほうが怖い。回帰テストにしておけば、将来の自分(と自己進化ジョブ)が壊したときに即座に赤くなります。

## ヒヤリハット集(隠さない)

このプロジェクトの信頼はガードの実話でしか成り立たないので、やらかしも書きます。

### `.claude/settings.json` をリポジトリにコミットしていた

開発に使っていた Claude Code の設定ファイル `.claude/settings.json` を、個人用の設定を含んだままリポジトリにコミットしていました。このファイルは **clone した人全員の Claude Code に適用されます**。エージェントの安全設計を語るリポジトリが、clone 先の開発環境の挙動を意図せず変えていた、という笑えない話です。

対応として、個人設定は `settings.local.json` へ移し、リポジトリ側に残すのは**安全側の deny 設定のみの最小構成**にしました。

### QRコードが黙って消えたバグ

リモート接続用のQRコードがある日、UIから黙って消えました。原因は localStorage キーのリネーム後、**旧キーのみを参照するフォールバックが残っていた**こと。新キーで保存されたデータを読めず、エラーも出さずに非表示になっていました。
