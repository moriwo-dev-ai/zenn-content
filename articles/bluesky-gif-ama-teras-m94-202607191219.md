## 導入

AMA-teras は Electron 製のデスクトップ AI コーディングエージェントです。

対話しながら、計画、実装、デバッグ、テストを進めます。  
足りない機能に出会うと、必要なツールや実装を生成し、それを取り込むところまで進めます。

ただし、生成したものがそのまま本体へ混ざるわけではありません。

AMA-teras では、生成物を隔離した `git worktree` で作り、検証ゲートを通し、最後に人間が承認してから本体へ昇格します。外部への発信や書き込みも、承認ダイアログを通さないと実行できない設計にしています。

この記事は、M94 で起きた「本体が自分で Bluesky の GIF 添付対応を実装した」ログです。

## 今回起きたこと

M94 では、Bluesky 投稿で GIF 添付に必要な対応が見つかりました。

そこで core 進化ジョブが実装を生成しました。  
生成された実装は、そのまま本体に混ぜず、進化ジョブとして扱いました。

今回昇格したのは次のジョブです。

- `job-42`
- `job-47`

この2つを昇格し、Bluesky GIF 添付に必要な対応を本体へ取り込みました。

ここで重要なのは、「AI がコードを書いた」ことだけではありません。  
むしろ重要なのは、AI が書いたコードが、勝手に本体へ混ざらないことです。

## 重要なのは“勝手に混ざらない”こと

AMA-teras の自己進化では、生成物を隔離した `git worktree` で作ります。

本体の作業ツリーへ直接書き込ませるのではなく、ジョブごとに分けた場所で変更を作ります。  
そのため、途中で壊れても本体へ即座に影響しません。

昇格前には検証ゲートを通します。

- typecheck
- テスト
- 実起動スモーク

これらを通したうえで、本体への昇格には人間の承認を必須にしています。

「自分で実装する」ことと、「自分で本体へ混ぜる」ことは分けています。  
AMA-teras の自己進化は、後者を自動化しきらない設計です。

## 途中で直した土台

M94 の周辺では、GIF 添付そのものだけでなく、自己進化を安全に回すための土台も直しました。

### `jobs.json` の引き裂き書き込みを防いだ

進化ジョブの状態を扱う `jobs.json` で、引き裂き書き込みを防ぐ対応を入れました。

対応は次の2つです。

- 書き込みの直列化
- アトミック置換

ジョブ管理のファイルが壊れると、実装そのものが正しくても、進化ジョブの状態管理が壊れます。  
自己進化では、生成コードだけでなく、ジョブの台帳も本体の一部です。

### core / renderer ジョブの `maxTurns` を増やした

core / renderer ジョブの `maxTurns` を変更しました。

- 変更前: `60`
- 変更後: `120`

進化ジョブが途中で打ち切られると、実装や修正が完了しないまま残ります。  
今回、core / renderer ジョブで使う上限を 120 に増やしました。

### Windows のテストフレークを潰した

Windows のテストフレークも潰しました。

自己進化の昇格条件にテストを入れている以上、テストが不安定だと、実装の良し悪しとは別の理由で昇格判断が揺れます。  
そのため、フレークを放置せず、昇格ゲートの信頼性を上げる対象として扱いました。

## 運営側の学び

今回の見せ場は、「AI が Bluesky GIF 添付を実装した」ことです。

ただ、運営側の学びとしては、それだけではありません。

自己進化の本体は、派手な自動実装だけではありません。  
壊れた状態を検知し、隔離し、検証し、最後に人が判断できる構造のほうが重要です。

過去のログでも、AMA-teras ではこの部分で何度も失敗しています。

たとえば、Bluesky では検索だけ 401 になる問題がありました。  
原因は app password ではなく、`BlueskyReader` の既定 `fetchImpl` が第2引数の `Authorization` ヘッダを捨てていたことでした。

ログインは別の既定経路を使っていたため成功し、検索だけが無認証で `bsky.social` に飛んでいました。  
テストではすべて `fetch` を注入していたため、本番だけが既定値を使う状態でした。

このときは、bluesky / zenn / hn / hatena の既定を修正し、注入なし経路をテストで固定しました。  
実機では巡回成功と検索10件を確認しました。

また、Zenn 公開まわりでは、承認経路や台帳の状態管理で複数の問題が出ました。

- スマホ承認が `bulkRespond` を通るため、重複まとめが効いていなかった
- 5秒間に3本 Zenn へコミットされ、同じ記事が2本公開リポジトリに載った
- `published: true` で push しても、Zenn 側では 403 のまま読めない記事があった
- 台帳が「実際に読めるか」ではなく「こちらが出したつもりか」を記録していた
- `DraftStore.list()` が読み込みのたびに `posted` を `staged` へ倒し直していた

これらはすべて、実装の問題であると同時に、運用上のガードの問題でもありました。

M94 の自己進化でも同じです。  
「コードを生成できた」だけでは足りません。  
生成物がどこで作られ、どの検証を通り、誰が最後に承認したのかが残る必要があります。

## まとめ

M94 では、Bluesky 投稿で GIF 添付に必要な対応が見つかり、core 進化ジョブが実装を生成しました。

その後、`job-42` と `job-47` を昇格しました。

ただし、AMA-teras の自己進化で主役にしたいのは、「自動で書く」ことだけではありません。

主役は次です。

- 隔離した `git worktree` で生成する
- typecheck・テスト・実起動スモークを通す
- 本体への昇格は人間の承認を必須にする
- 外部への発信・書き込みも承認ダイアログを通す
- 壊れた状態を検知し、台帳やジョブ管理の土台も直す

AMA-teras の自己進化は、「自動で書く」よりも「安全に昇格する」ことを中心に置いています。

次に公開するログでも、成功した実装だけではなく、ガードや失敗の実話を出します。
