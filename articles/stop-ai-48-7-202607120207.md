## TL;DR

- `GodScheduler.stop()` が `clearInterval` しかしておらず、起動遅延用の `setTimeout` を止め忘れていた
- 設定保存のたびに `operations.reset()` → 再initが走り、停止したはずの旧インスタンスが90秒後に蘇って増殖した
- 2026-07-12 00:00〜00:01 の48秒間に、神議(AI会議)が7回連続実行(各2.3〜3.1kトークン、計約19k)、承認バッチが6枚生成され、手で直したはずの `schedule.json` が古い状態に巻き戻った
- 修正の核心は「タイマーを止める」ことではなく、「**停止済みインスタンスは書き込まない**」を構造で保証したこと

## AMA-terasの自律運営とスケジューラ

### 神々の時計: 観測・巡回・下書き・神議を定期実行する仕組み

AMA-terasには「神々の時計」(AMENO-koyane)と呼ぶ運営スケジューラがあり、観測・巡回・下書き生成・トリアージ・神議(AIによる運営会議)を定期実行している。稼働/停止や1日予算はUIから操作でき、状態は `schedule.json` に永続化される。

外部への発信は岩戸ゲート(承認ダイアログ)を通るため、スケジューラが暴走しても勝手に投稿されることはない。だが、**LLMを呼ぶこと自体はスケジューラの裁量**であり、暴走すればトークンは消費される。今回の事件はまさにそこで起きた。

### なぜ起動90秒後に setInterval を張る設計だったか

`GodScheduler.start()` は、起動直後にいきなり `setInterval` を張るのではなく、**「90秒後に `setInterval` を張る `setTimeout`」** という二段構えのタイマーを使っていた。概念的にはこうだ。

```typescript
// 修正前(概念コード)
start() {
  setTimeout(() => {
    this.timer = setInterval(() => this.tick(), INTERVAL_MS);
  }, 90_000); // ← このsetTimeoutへの参照を保持していなかった
}

stop() {
  clearInterval(this.timer); // ← setIntervalしか止めていない
}
```

この「二段目のタイマーだけを止めている」構造が、今回の事件のすべてだった。

## 事件: 2026-07-12 00:00の48秒間

### 症状1: 神議7回連続実行(各2.3〜3.1kトークン)と承認バッチ6枚

2026-07-12 00:00〜00:01 の48秒間に、神議が**7回連続で実行**された。1回あたり2.3〜3.1kトークン、合計約19kトークン。承認バッチも6枚生成された。

神議は本来、定期実行の1サイクルに1回走るものだ。48秒に7回は明らかに異常だった。

### 症状2: 手で直したはずの schedule.json が古い状態に巻き戻る

さらに不気味だったのがこちら。M38で `schedule.json` を直接編集して観測神 omoi-kami を再開(enabled=true / 予算20000 / 🔒)していたのだが、**そのファイルが編集前の古い状態に巻き戻っていた**。

ファイルを直したのに、誰かが古い内容で上書きしている。ディスク上のデータと、メモリ上のどこかの状態がずれている——そう疑わせる症状だった。

### 実データで発覚した——テストでは捕まらなかった理由

このバグは既存テストでは捕まらず、**実データで発覚した**(PROGRESS.mdにも「実害あり・実データで発覚」と記録している)。

再現条件が「`start()` → `stop()` → 90秒経過」という時間依存のシーケンスであり、しかもトリガーが「設定保存」という一見無関係な操作だったためだ。`stop()` を呼んだ直後にスケジューラが止まっていることを確認するテストでは、90秒後の復活は検出できない。

## 原因: setTimeout の中の setInterval

### stop() は clearInterval しかしていなかった

前掲のとおり、`start()` は `setTimeout` → `setInterval` の二段構えだが、`stop()` は `clearInterval` しかしていなかった。つまり、**まだ `setInterval` が張られる前(起動から90秒以内)に `stop()` を呼ぶと、何も止まらない**。90秒後、`setTimeout` のコールバックが発火し、停止したはずのインスタンスが `setInterval` を張って動き出す。

### 設定保存 → operations.reset() → 再init のたびに幽霊が1体増える

これが単発なら被害は小さかった。問題は、設定保存(`secretsSet` 等)のたびに `operations.reset()` が走り、次アクセスで再initされる設計だったこと。

つまり:

1. 設定を保存する
2. `operations.reset()` で旧スケジューラが `stop()` される(が、`setTimeout` は生きている)
3. 次アクセスで新スケジューラがinitされ `start()` する
4. 90秒後、**旧スケジューラが蘇る**。新スケジューラと合わせて2体に
5. また設定を保存すると3体に……

設定保存のたびに幽霊が1体ずつ増える。48秒間に神議が7回走ったのは、増殖した幽霊スケジューラたちがそれぞれ神議を実行していたからだ。

### 全幽霊が同じ schedule.json を古いメモリ状態で上書き

`schedule.json` の巻き戻りもこれで説明がつく。各幽霊は自分がinitされた時点の状態をメモリに持ったまま蘇り、**同じ `schedule.json` に古いメモリ状態を書き戻す**。手動編集でomoi-kamiを復活させても、それを知らない幽霊が保存処理を走らせるたびに巻き戻る。

## 修正: 止めるのではなく「書けなくする」

### startTimer を保持して clearTimeout / stopped フラグで復活を禁止(使い捨て設計)

まず素直な修正として、`setTimeout` の返り値(`startTimer`)を保持し、`stop()` で `clearTimeout` するようにした。加えて `stopped` フラグを立て、**一度stopしたインスタンスは二度とstartできない使い捨て設計**にした。復活の余地を残さない。

### tick() と save() を stopped で早期return——巻き戻しの構造的防止

しかし本命はこちら。`tick()` と `save()` の先頭に `stopped` チェックを入れ、**停止済みインスタンスからの実行・書き込みを構造的に禁止**した。

```typescript
// 修正後(概念コード)
tick() {
  if (this.stopped) return; // 幽霊は実行できない
  // ...
}

save() {
  if (this.stopped) return; // 幽霊は書き込めない
  // ...
}
```

タイマーの止め忘れは「またやる」可能性のあるミスだ。だが、たとえタイマーの管理に将来また穴が開いても、**停止済みインスタンスがファイルに触れない限り、状態の巻き戻しは起きない**。「止める」ではなく「書けなくする」——防御の層を一段深いところに置いた。

### 偽タイマーによる再現テスト4件

偽タイマー(fake timers)を使って「`stop()` 後に90秒進めても蘇らない」ことを含む4件のテストを追加し、この挙動を固定した。時間依存のバグだからこそ、時間を操作できるテストで再現・固定する。
