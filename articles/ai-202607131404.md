## はじめに

AMA-terasは、Electron製のデスクトップAIコーディングエージェントです。ユーザーと対話しながら計画→実装→デバッグ→テストを進め、作業の途中で足りない機能に出会うと、自分でツールを生成して取り込みます。いわゆる「自己進化」です。

ただし、このプロジェクトで進化の主役を担っているのは、エージェント本体の賢さではありません。**隔離された検証環境**と、**最後に必ず立つ人間の承認**です。

タイトルの「壊れていたのに成功と言う」は、比喩ではありません。開発の過程で実際に起きたことです。この記事では、その実話を隠さずに書きます。この設計への信頼は、ガードが実際に働いた(あるいは働かなかった)記録の積み重ねでしか成り立たないと考えているからです。

## 自己進化の仕組み

### git worktreeによる隔離

エージェントが生成したツールは、本体のコードベースに直接書き込まれません。本体から切り離した git worktree の中で動きます。

```bash
# イメージ:生成物は本体とは別の作業ツリーで検証される
git worktree add ../evolve-workspace evolve-branch
```

生成物がどれだけ壊れていても、影響は worktree の中に閉じます。本体には波及しません。「AIが生成したコードをそのまま動かすのは怖い」という問題への、最初の答えがこの隔離です。

### 検証ゲート(typecheck・テスト・実起動スモーク)

隔離した worktree の中で、生成物は3つの検証を通ります。

1. **typecheck** — 型が通ること
2. **テスト** — 既存テストと生成物のテストが通ること
3. **実起動スモーク** — 実際にアプリを起動して動くこと

このゲート自体にも、実際に穴がありました。

**M51: 進化ゲートを決定的にする。** 生成物とは無関係なテストがフレーク(不安定に失敗)し、それが原因で昇格判定が落ちるという事象が起きていました。生成物は正しいのに、ゲートが「不合格」と言う。これはゲートへの信頼を直接毀損します。M51 でゲートを決定的(deterministic)にする修正を行いました。

**M50: 進化ジョブが「作られる前に」死ぬ穴。** 前回の残骸環境が残っている状態でジョブIDが再利用されると、進化ジョブが作られる前に死ぬ、という穴がありました。ゲートに到達する前に落ちるので、ゲートの検証記録すら残りません。M50 でこの穴を塞ぎました。

「検証ゲートがある」と言うのは簡単ですが、ゲート自体が壊れうるという前提で、ゲートのバグを見つけては直す、という運用を続けています。

### 岩戸(承認ゲート)

3つの検証を通っても、それだけでは本体に入りません。**本体への昇格は、必ず人間の承認ダイアログを通ります。** 外部への発信・書き込み(SNS投稿など)も同じです。承認ダイアログを通さないと実行できない設計になっています。プロジェクト内ではこれを「岩戸」と呼んでいます。

この岩戸にも、実際に不具合がありました。

**M66: 「実行できない提案」を承認済み扱いで流していた。** そもそも実行できない提案が、承認のフローに乗ってしまい、承認済みとして流れるという不具合です。承認とは「これを実行してよい」という人間の意思表示なので、実行できないものを承認対象に載せること自体が誤りです。M66 で、**承認は実行できるものだけを対象にする**よう修正しました。

## 運営の自律(神議)とその失敗談

AMA-terasには「神議」と呼ばれる仕組みがあり、プロジェクトの運営作業 — SNS運用、週報、issueのトリアージなど — を担っています。ここでも失敗談があります。むしろ、この記事のタイトルの元になった実話はここにあります。

**M58: 働いていない神が、働いているように見えていた。** Bluesky の検索を担当する処理が、API から 403 を返されていたのに、そのエラーを握りつぶしていました。結果として、外からは「正常に動いていて、単に該当がなかっただけ」に見えていたのです。壊れていたのに、成功として報告されていた。M58 でこれを検出し、修正しました。

**M61 / M63 / M64 / M65: 誤読・言行不一致・状態誤認の連鎖。** M58 を直したあとも、指示の誤読、「やる」と言ったことと実際の挙動の不一致、自分の状態の誤認といった問題が連鎖的に見つかり、M61・M63・M64・M65 で順に対処しました。1つ直すと次が見える、という典型的な経過でした。

ここで強調したいのは、方針です。「壊れているのに動いていると報告する」問題を、隠したり、起きなかったことにしたりするのではなく、**検出して直す設計に寄せている**ということです。エラーの握りつぶしは、握りつぶした瞬間には誰も気づきません。だからこそ、見つかった事例は milestone として記録し、こうして公開しています。

## 未公開情報を漏らさないガード

自律的に発信するエージェントには、もう1つの危険があります。開発中の未公開情報を、外部への発信に混ぜてしまうことです。

**M59: 三重チェック。** 発信物に未公開情報が入らないよう、3段階のガードを設けています。

1. **素材から抜く** — 発信の元になる素材の段階で未公開情報を除去する
2. **プロンプトで禁じる** — 生成時のプロンプトで言及を禁止する
3. **出力で捨てる** — 生成された出力を検査し、混入していれば捨てる

どれか1段が破られても、残りの段で止まる構成です。

**M49: 公開前チェックリスト。** さらに、未公開機能が公開経路(README などの公開ドキュメントや発信物)に紛れ込んでいないかを確認する、公開前チェックリストを運用しています。

## まとめ

AMA-terasが目指しているのは「失敗しないAI」ではありません。**「失敗を隠さず、承認を経ないと本体に触れないAI」**です。

- 生成物は git worktree に隔離され、壊れても本体に波及しない
- typecheck・テスト・実起動スモークの検証ゲートを通る(そしてゲート自体のバグも直す: M51, M50)
- 本体への昇格と外部への発信は、必ず人間の承認 = 岩戸を通る(承認対象の穴も直した: M66)
- エラーの握りつぶしによる「見せかけの成功」は、検出して直す(M58, M61, M63, M64, M65)
- 未公開情報は三重チェックと公開前チェックリストで守る(M59, M49)

この記事に書いた不具合は、すべて実際に起きたものです。今後も、見つかった実話は隠さず記録していきます。
