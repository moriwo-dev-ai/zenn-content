## はじめに

AMA-terasは、Electron製のデスクトップAIコーディングエージェントです。ユーザーと対話しながら計画→実装→デバッグ→テストを進め、途中で「この機能が足りない」と分かると、自分でツールを生成してプロジェクトに取り込みます。私たちはこれを「自己進化」と呼んでいます。

「自己進化」と聞くと、AIが勝手に自分を書き換えていく姿を想像するかもしれません。しかしAMA-terasの進化の主役は本体ではありません。**隔離された検証環境**と、**最後に必ず立つ人間の承認**です。

タイトルの「壊れていたのに成功と言う」は、開発中に実際に起きた問題です。この記事では、その実話を含めて、自己進化を支えるガードの仕組みと、それが破られかけた失敗談を紹介します。

## 自己進化の仕組み

### git worktreeによる隔離

AMA-terasが生成したツールは、いきなり本体に取り込まれません。まず本体から切り離した**git worktree**の中で動かします。

生成物がどれだけ壊れていても、worktreeの中で壊れるだけです。本体には波及しません。「AIが自分のコードを壊して動かなくなる」という自己進化の最悪シナリオを、構造で潰しています。

### 検証ゲート(typecheck・テスト・実起動スモーク)

worktree内の生成物は、本体への昇格の前に検証ゲートを通ります。

- **typecheck**: 型が通ること
- **テスト**: 既存テストと生成物のテストが通ること
- **実起動スモーク**: 実際にアプリを起動して動くこと

このゲート自体にも不具合がありました。

**M51: 進化ゲートを決定的にする。** 生成物とは無関係なテストがフレーク(不安定に落ちたり通ったりする状態)を起こし、昇格判定が落ちていました。生成物は正しいのに「検証失敗」と判定される。逆に言えば、フレークが通れば壊れたものが通りかねない。判定は決定的でなければ意味がありません。

**M50: 進化ジョブが「作られる前に」死ぬ穴。** 過去の進化の残骸環境が残っている状態でIDが再利用されると、進化ジョブが作られる前に死ぬケースがありました。この穴を塞ぎました。ゲートは「通す・落とす」の判定だけでなく、「そもそもゲートに到達できるか」まで含めて信頼できる必要があります。

### 岩戸(承認ゲート)

検証ゲートを通っても、それだけでは本体に入れません。**本体への昇格、外部への発信・書き込みは、必ず人間の承認ダイアログを通します**。この承認ゲートを「岩戸」と呼んでいます。

この思想はコーディング以外の機能にも一貫しています。たとえば音声から約束・ToDoを抽出する機能(月読モード)でも、抽出結果を記録に勝手に書き込まず、確認UIで人が承認したものだけを残す設計です。話者識別を持たないため「同席者の発言を本人の約束と取り違える可能性が残る。だから確認UIが必須」という判断です。

岩戸にも不具合がありました。

**M66: 「実行できない提案」を承認済み扱いで流していた。** 承認ダイアログに、そもそも実行できない提案が乗り、承認済みとして流れていました。承認は「実行できるもの」だけを対象にするよう修正しました。人間の承認は最後の砦なので、承認の対象が信頼できないと砦ごと無意味になります。

## 運営の自律(神議)とその失敗談

AMA-terasには「神議」と呼ぶ、プロジェクト運営(SNS運用・週報・トリアージなど)を自律的に担う仕組みがあります。ここでも失敗が続きました。

**M58: 働いていない神が、働いているように見えていた。** Bluesky検索が403を返していたのに、そのエラーを握りつぶしていました。結果、検索は一切機能していないのに、外からは正常に動いているように見えていた。まさに「壊れているのに動いていると報告する」状態です。

**M61 / M63 / M64 / M65:** その後も、誤読・言行不一致・状態誤認が連鎖して見つかりました。1つの誤認を直すと、その裏に隠れていた次の誤認が見えてくる、という展開です。

これらから得た方針は明確です。「壊れているのに動いていると報告する」問題は、隠すのではなく**検出して直す**設計に寄せる。エラーを握りつぶさず、状態を誤認したまま報告させない。失敗談をこうして記録し公開しているのも、同じ方針の一部です。

## 未公開情報を漏らさないガード

自律的に外部発信する仕組みを持つ以上、未公開情報の漏洩は構造で防ぐ必要があります。

**M59: 三重チェック。** 発信物に対して、

1. **素材から抜く**: 未公開情報を素材の段階で除去する
2. **プロンプトで禁じる**: 生成時に言及を禁止する
3. **出力で捨てる**: 出力段階でチェックして落とす

の三重チェックを敷いています。どれか1段が破られても残りが止める構えです。

**M49: 公開前チェックリスト。** 未公開機能が公開経路に紛れ込まないよう、公開前チェックリストを運用しています。実例として、未公開の月読モード開発時には、README / USER-GUIDE / RELEASE / CONTRIBUTING / ARCHITECTURE / remote-ui のいずれにも記載が増えていないことをgrepで確認し(0件)、関連イベントをリモートUIへのSSE中継対象に入れないことをテストで固定しています。「漏れていないこと」自体をテストで守る形です。

## まとめ

AMA-terasが目指しているのは「失敗しないAI」ではありません。**「失敗を隠さず、承認を経ないと本体に触れないAI」**です。

- 生成物はgit worktreeで隔離され、壊れても本体に波及しない
- typecheck・テスト・実起動スモークの検証ゲートを通る(そしてゲート自体の穴も直してきた)
- 本体への昇格と外部への発信・書き込みは、必ず人間の承認(岩戸)を通る
- エラーの握りつぶしや状態誤認は、隠すのではなく検出して直す
- 未公開情報は三重チェックと公開前チェックリストで守る

この記事に書いた不具合(M50, M51, M58, M59, M61, M63〜M66)はすべて実際に起きたことです。今後も見つかった実話は隠さず記録していきます。この製品の信頼は、ガードが完璧であることではなく、ガードが破られかけた実話を公開し続けることで成り立つと考えているためです。
