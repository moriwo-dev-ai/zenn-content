## 導入: AMA-terasが目指しているもの

AMA-teras は、Electron 製のデスクトップ AI コーディングエージェントです。

対話しながら、次の流れを進めます。

1. 計画する
2. 実装する
3. デバッグする
4. テストする
5. 必要ならツールを増やす

ただし、AMA-teras が目指しているのは「AI が全部勝手にマージする」ことではありません。

足りない機能に出会ったとき、AMA-teras は実装案を出し、必要なツールや core を生成し、検証に回します。  
しかし、本体へ昇格する最後の判断は人間が行います。

生成物は隔離した git worktree で検証します。  
typecheck、テスト、実起動スモークを通し、最後に人間の承認ゲートを通します。

AMA-teras で大事にしているのは、「成功しました」という自己申告をそのまま信用しないことです。

外の世界に本当に出たのか。  
ユーザーが本当に読めるのか。  
ボタンは本当に押せる状態に出ているのか。  
台帳は事実を書いているのか。

この記事では、直近の実装ログに残っている失敗と修正をもとに、その設計を紹介します。

## 見せ場1: 自己進化はできた。でも自動マージではない

AMA-teras には、足りない機能に出会ったら自分でツールを生成して取り込む、という自己進化の考え方があります。

ただし、ここでいう自己進化は「自動マージ」ではありません。

生成物は本体とは分けて扱います。  
隔離した git worktree で検証し、最低限次のゲートを通します。

- typecheck
- テスト
- 実起動スモーク
- 人間による昇格承認

この最後の人間承認を、AMA-teras では岩戸ゲートとして扱っています。

AI が「できました」と言っても、それだけでは本体に入りません。  
実際にビルドできること、テストが通ること、起動できることを確認し、そのうえで人間が昇格を承認します。

自己進化を語るとき、成果物だけを見せると危険です。

重要なのは、次の2つをセットで見ることです。

- 何を生成したか
- どのガードを通って本体に入ったか

AMA-teras では後者を省略しません。

## 見せ場2: 必要な機能に出会ったら、道具から増やす

直近では、Zenn 投稿まわりで「アプリからできること」と「実際に必要な運用」のズレが続けて見つかりました。

M73 では、Zenn の「記事化」が `published: false` でコミットするところまでで止まっていました。  
つまり、記事ファイルはできているのに、そこから先をアプリから公開できませんでした。

そこで `zenn-repo` に `publish-article` を追加しました。

流れは次の通りです。

1. 岩戸ゲートで全文を確認する
2. `published: true` にする
3. push する

ただし、公開条件を満たさない記事は `blocked` として扱い、承認を求める段階に進めません。

ここでも「機能を足した」だけでは終わっていません。

公開操作は外部への書き込みです。  
そのため、アプリが勝手に実行するのではなく、人間の承認ダイアログを通します。

AMA-teras の自己進化は、道具を増やすことだけではなく、その道具をどの承認経路に乗せるかまで含みます。

## 見せ場3: 「壊れていたのに成功を報告していた」失敗

M76 では、Zenn 投稿で大きな失敗がありました。

3本の記事を `published: true` にして push しました。  
しかし、Zenn が同期したのは2本だけでした。

残り1本は Zenn 上では 403 でした。  
存在はしているが、誰も読めない状態です。

それにもかかわらず、アプリの台帳は間違った状態を記録していました。

- 実際に公開された2本を「公開待ち」のまま残した
- Zenn が同期していない1本を「投稿済み」と記録した

つまり、台帳は世界の実状態ではなく、こちらの申告を書いていました。

`published: true` にした。  
push した。  
だから投稿済み。

この判断が間違っていました。

修正では、Zenn に一次情報を聞くようにしました。

`ZennReader.isLive(slug)` を追加し、結果を次のように扱います。

- 200: 読める
- 403 / 404: 読めない
- 不達: 不明

そして台帳は、実際に読めるものだけを `posted` にします。

push しただけでは「公開した」と言いません。  
Zenn で読めることを確認してから、投稿済みとして扱います。

その後の M78 でも、同じ系統の問題が残っていました。

M76 で「Zenn で実際に読めるものだけ `posted`」にしたにもかかわらず、読み込み時の古い変換処理が `posted` を `staged` に倒し直していました。

原因は、過去の前提が残っていたことです。

当時は「アプリからは公開できないのだから `posted` はありえない」という前提がありました。  
しかし M73 でアプリから Zenn を公開できるようになり、M76 で Zenn に実状態を聞くようになりました。  
前提は消えたのに、変換だけが残っていました。

修正では `migrateStaged` を撤去しました。  
下書きの状態を憶測で倒さず、公開されたかどうかは Zenn に聞きます。

実機確認では、台帳が一次情報と一致しました。

- Zenn で 200 の3本: `posted`
- 403 の1本: `staged`

神議も「公開待ち4本」とは言わなくなり、本当に残っている問題だけを指すようになりました。

## 見せ場4: スマホ承認で5秒間に3本コミットした話

M74 では、スマホ承認で重複投稿が起きました。

実害として、23:05:05、23:05:07、23:05:10 の5秒間に3本、Zenn へコミットされました。  
その結果、同じ記事が2本、公開リポジトリに載りました。

原因は、重複ガードの経路漏れです。

M69 で「双子まとめ」は入っていました。  
しかし、それは `batchRespond` にしかありませんでした。

一方で、スマホ承認は1枚でも `bulkRespond` を通ります。  
つまり、PC 側で効いていた重複ガードが、スマホ経路では効いていませんでした。

さらに、下書きの「投稿済み」印は実行完了後に付きます。  
本文生成の数秒間に、同じカードを続けて叩くと、競合で重複ガードをすり抜けられました。

修正は2つです。

- `bulkRespond` を直列化する
- `bulkRespond` 側にも双子まとめを入れる

この失敗で分かったのは、「同じ操作に見えるもの」が同じコード経路を通るとは限らない、ということです。

PC で直したからスマホでも直っている、とは言えません。  
実際の承認経路ごとにガードを置く必要があります。
