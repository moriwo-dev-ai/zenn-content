## 導入

AMA-teras は Electron 製のデスクトップ AI コーディングエージェントです。

対話しながら、計画、実装、デバッグ、テストを進めます。生成物は隔離した git worktree で typecheck、テスト、実起動スモークの検証ゲートを通し、本体へ昇格する前には人間の承認を必ず挟みます。

外部への発信や書き込みも同じです。承認ダイアログを通さないと実行できない設計にしています。

ただし、今回直したのは「承認したか」ではありません。

問題はその先でした。

```text
承認した
push した
published: true にした
```

これらは、どれも「本当に読める」ことを意味しません。

AMA-teras が目指しているのは、“自律”そのものではなく、“確認できる自律”です。  
そのためには、成功ログではなく、外部サービス側の一次情報を見なければいけません。

今回の記事は、Zenn と GitHub Release の公開台帳で起きた失敗と、その修正の記録です。

## 失敗1: Zennへpushしたのに誰も読めなかった

M76 で起きた問題です。

AMA-teras から Zenn 記事を公開する流れでは、岩戸ゲートで全文を確認し、承認後に `published: true` にして push します。

しかし、実際にはこれだけでは不十分でした。

3本の記事を `published: true` にして push したところ、Zenn が同期したのは2本だけでした。残り1本は Zenn 側で 403 でした。

PROGRESS にはこう残っています。

```text
403(存在するが非公開)= 誰も読めない
```

つまり、リポジトリ上では `published: true` になっていても、Zenn 側で 403 なら露出はゼロです。

ところが当時の台帳は、外の世界の状態ではなく「こちらの申告」を記録していました。

その結果、台帳は逆向きに壊れました。

```text
実際に公開された2本:
  「公開待ち」のまま残る

Zennが同期していない1本:
  「投稿済み」と記録される
```

実際に読める記事が「公開待ち」として残り、誰も読めない記事が「投稿済み」になる。  
公開台帳としては一番まずい状態です。

修正は、Zenn 側の一次情報を見るようにすることでした。

`ZennReader.isLive(slug)` を追加し、記事 URL が実際に読めるかを確認します。

判定は次の通りです。

```text
200       = 読める
403 / 404 = 読めない
不達      = 不明
```

そして、台帳では「200 で実際に読めるものだけ」を `posted` として扱うようにしました。

ここで重要なのは、公開ボタンの意味も変えたことです。

```text
pushしただけで「公開した」とは言わない
```

承認され、push されたことと、Zenn 上で読めることは別の事実として扱います。

## 失敗2: 次の読み込みが真実を上書きしていた

M76 で「実際に読めるものだけを `posted` にする」修正を入れました。

しかし、M78 で別の問題が見つかりました。

Zenn で実際に読める記事を `posted` にしても、次に読み込むとまた `staged` に戻っていたのです。

原因は M57 時代の前提でした。

当時は「アプリからは公開できない」という前提がありました。  
そのため、`DraftStore.list()` には次のような変換が残っていました。

```text
zenn / github の下書きが posted なら
読み込みのたびに staged へ倒し直す
```

当時の前提では、アプリから公開できないのだから `posted` はありえない、という判断でした。

しかし、その前提はすでに消えていました。

Zenn については、アプリから公開操作を行うようになっていました。  
さらに、Zenn 側に実際に読めるかを聞くようにもなっていました。

にもかかわらず、古い変換だけが残っていました。

結果として、一次情報に基づいて正しい状態を書いても、次の読み込みでアプリ側の憶測がそれを上書きしていました。

修正では、`migrateStaged` を撤去しました。

```text
下書きの状態を憶測で倒さない
公開されたかどうかは一次情報に聞く
```

この挙動は `m78.test.ts` で固定しました。

確認した内容は、`posted` な zenn / github 下書きが `list()` を跨いでも `posted` のままであることです。

実機確認では、台帳が一次情報と一致しました。

```text
Zenn 200 の3本:
  posted

Zenn 403 の1本:
  staged
```

これで、内部の提案ロジックも「公開待ち4本」とは言わなくなりました。  
代わりに、`published: true` なのに Zenn 側で露出ゼロになっている記事だけを、本当の問題として指すようになりました。

## 失敗3: GitHub Releaseでも同じ病気が出た

同じ種類の問題は GitHub Release 側にも残っていました。

M87 の話です。

v1.2.0 の下書きリリースを作成し、公開したあとも、台帳がそれを「公開待ち」と言い続けました。

Zenn で起きた問題と同じです。  
台帳が外部サービス側の状態を見ていませんでした。

ただし、GitHub Release 側にはもう一つ問題がありました。

```text
どの下書きがどのリリースなのか突き合わせる鍵が無かった
```

下書きはタグを覚えていませんでした。  
そのため、GitHub 側に公開済みのリリースがあっても、アプリ内のどの下書きと対応するのかを正しく判断できません。

修正では、`OperationsDraft.tag` を追加しました。

リリース作成時にタグを保持し、そのタグを使って GitHub 側のリリースと突き合わせます。

そして、GitHub 側の一次情報を見ます。

```text
draft=false
```

GitHub Release が `draft=false` なら、世に出ているものとして台帳を更新します。

更新タイミングは2つです。

```text
公開した瞬間
releaseInfo のポーリング時
```

`releaseInfo` は 30 秒ごとに取り直すようにしました。  
取得できないときは、憶測で状態を倒しません。

この点も Zenn 側の修正と同じです。

```text
引けないときは倒さない
一次情報が取れたときに更新する
```

v1.2.0 は 2026-07-14 14:39 JST に公開済みで、インストーラ添付1本の状態になりました。

## 承認ゲートとの関係

AMA-teras では、外部公開は岩戸ゲートで人間が確認します。

Zenn 記事の公開では、全文確認のあとに `published: true` へ変更し、push します。  
GitHub Release でも、公開操作は人間の承認を通します。

これは必要なガードです。

ただし、今回の失敗で分かったのは、承認ゲートだけでは公開台帳として足りないということです。

承認ログが示すのは、あくまで次の事実です。

```text
人間が内容を確認した
人間が実行を承認した
アプリが実行を試みた
```

一方で、外の世界に届いたかどうかは別の事実です。

```text
Zennで200が返る
GitHub Releaseがdraft=falseになっている
```

この2つは分けて扱う必要があります。

承認ログを見れば、「勝手に外へ出していない」ことは確認できます。  
一次情報を見れば、「本当に外で見える状態になっている」ことを確認できます。

片方だけでは足りません。

今回の修正では、公開台帳を「実行済みログ」ではなく「外部サービス上の状態に寄せる台帳」に直しました。

## まとめ

今回の修正で一貫していた問題は、成功ログを信じすぎたことでした。

```text
published: true にした
push した
公開ボタンを押した
承認された
```

これらは重要な事実です。  
しかし、「本当に読める」「本当に公開されている」とは限りません。

Zenn では `ZennReader.isLive(slug)` で 200 のものだけを `posted` にしました。  
読み込み時に憶測で `posted` を `staged` へ戻す変換も撤去しました。  
GitHub Release では `OperationsDraft.tag` を保持し、GitHub 側の `draft=false` を見て台帳を更新するようにしました。

外部公開には人間の承認が必要です。  
そのうえで、承認後も一次情報を確認する必要があります。

失敗談を隠さないのは、AMA-teras が「何でも自動でやる」ことではなく、「何が起きたか確認できる」ことを重視しているからです。

公開台帳に必要なのは、こちらの成功報告ではありません。

```text
世界の側で、実際にどう見えているか
```

それを見るように直した、という話でした。
