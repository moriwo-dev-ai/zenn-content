## 導入: 出したつもりと、出ていることは別

AMA-terasでは、Zenn記事やGitHub Releaseの公開操作も、人間の承認を通して実行します。

ただし、今回の問題は「承認を通したか」ではありませんでした。

問題は、アプリが「公開したつもり」になっていたことです。  
`published: true` にしてpushした。処理は成功した。だから公開済みとして扱う。

しかし、実際にはZenn上で読めない記事がありました。  
逆に、実際には読める記事を、AMA-teras側の台帳が「公開待ち」として残していました。

このズレによって、自律運営側の判断も間違いました。  
神議は、間違った台帳を見て「公開待ちの記事がある」と催促し続けました。

今回直したのは、公開処理そのものではなく、公開後に「世界の状態」を確認して台帳を直す仕組みです。

## 何が起きたか

### Zennへpushしたのに、1本は403で誰も読めなかった

Zenn記事3本を `published: true` にしてpushしました。

しかし、Zennが同期したのは2本だけでした。  
残り1本はZenn上で403でした。

この状態は、ファイル上は `published: true` です。  
しかし、Zenn上では読めません。

つまり、アプリ側の操作としては「公開した」ように見えても、実際の読者からは読めない状態でした。

PROGRESSではこの状態を次のように記録しています。

- 3本を `published: true` にしてpush
- Zennが同期したのは2本
- 残り1本はZennで403
- 403は「存在するが非公開」で、誰も読めない

後続でZennから「投稿数の上限に達したためデプロイされませんでした」という通知も確認しました。  
このため、`published: true` なのに世に出ていない記事が残りました。

### 台帳は逆に「読める記事」を公開待ちに残していた

台帳のズレは、読めない記事だけではありませんでした。

実際にZennで読めるようになった2本を、AMA-teras側の台帳は「公開待ち」のまま残していました。

一方で、Zenn上で403の1本は「投稿済み」と記録していました。

つまり台帳は、現実と逆方向にズレていました。

| 実際の状態 | 台帳の状態 |
| --- | --- |
| Zennで読める2本 | 公開待ち |
| Zennで403の1本 | 投稿済み |

この状態では、台帳を見ても「何が本当に公開されているか」は分かりません。

### 神議が間違った状態をもとに催促し続けた

AMA-terasの神議は、台帳を見て次の行動を提案します。

その台帳が間違っていたため、神議は公開済みの記事を含む「公開待ち4本」を数え続けました。

実際には、Zennで読める記事はすでにありました。  
本当に問題だったのは、`published: true` なのにZenn側で露出していない記事です。

しかし、台帳が世界の状態と一致していなかったため、神議は「公開待ちの記事がある」と催促し続けました。

実機確認後は、台帳が一次情報と一致しました。

- Zennで200の3本 → `posted`
- Zennで403の1本 → `staged`

その結果、神議は「公開待ち4本」とは言わなくなり、`published: true` なのにZenn側で露出ゼロの記事だけを問題として指すようになりました。

## 原因

### 台帳が世界の状態ではなく、アプリ側の申告を書いていた

原因は、台帳が「世界で実際に起きていること」ではなく、「アプリ側で実行したこと」を記録していたことです。

`published: true` にした。  
pushした。  
だから公開済み。

この考え方では、Zenn側の同期失敗や403を扱えません。

公開処理が成功しても、公開先で読めるとは限りません。  
外部サービスが関わる以上、最終的な状態は外部サービスに聞く必要があります。

今回のケースでは、Zenn上で200が返るかどうかが一次情報でした。

### `published:true` と「実際に読める」は同じではなかった

Zenn記事のファイルに `published: true` が付いていても、それは「公開したい」というリポジトリ上の状態です。

読者が実際に読めるかどうかは、Zenn側の同期結果で決まります。

今回、少なくとも次の状態がありました。

- `published: true` でpush済み
- しかしZennでは403
- つまり誰も読めない

この状態を `posted` と扱うと、台帳上は完了になります。  
しかし、実際には露出ゼロです。

そこで、`published: true` と「実際に読める」を分けて扱う必要がありました。

### 過去の前提変換が残り、postedをstagedへ戻していた

M76で、Zennに実際に読めるか聞くようにしました。  
しかし、それでも神議は公開済みの記事を含む「公開待ち4本」を数え続けました。

原因は、過去の前提変換でした。

`DraftStore.list()` が、Zenn/GitHubの下書きが `posted` なら、読み込みのたびに `staged` へ戻していました。

これは、当時の前提に基づく変換でした。

> アプリからは公開できないのだから、`posted` はありえない

しかし、その前提は変わっていました。

- Zenn記事はアプリから公開できるようになった
- Zennに実際に読めるか確認するようになった

それでも変換だけが残っていたため、せっかく正しい状態を書いても、次の読み込みで `staged` に戻されていました。

## 直したこと

### `ZennReader.isLive(slug)` を追加

Zenn上で記事が実際に読めるか確認するために、`ZennReader.isLive(slug)` を追加しました。

扱いは次の通りです。

| Zennの応答 | 扱い |
| --- | --- |
| 200 | 読める |
| 403 | 読めない |
| 404 | 読めない |
| 不達 | 不明 |

ここで重要なのは、push成功をもって公開済みにしないことです。

台帳は、Zennで実際に読めるものだけを `posted` にします。

### 200だけをposted、403/404は読めないものとして扱う

`published: true` の記事であっても、Zennで403または404なら `posted` にはしません。

今回の実機確認では、次の状態になりました。

- Zenn 200の3本 → `posted`
- Zenn 403の1本 → `staged`

これにより、台帳は「公開操作をしたか」ではなく「実際に読めるか」に揃いました。

公開ボタンも、pushしただけで「公開した」とは言わないようにしました。

### `migrateStaged` を撤去し、憶測で状態を倒さない

読み込み時に `posted` を `staged` へ戻していた変換を撤去しました。

対象は `migrateStaged` です。

理由は単純です。  
現在は、公開されたかどうかをZennに聞けます。

であれば、古い前提で台帳を倒す必要はありません。

この修正はテストでも固定しました。

- `m78.test.ts`
- `posted` なZenn/GitHub下書きが `list()` を跨いでも `posted` のままであること

これで、「正しい状態を書いたのに、次の読み込みで壊れる」問題を止めました。

### GitHub Release側にもタグを持たせ、draft=falseを一次情報にした

同じ種類の問題は、GitHub Release側にもありました。

公開したリリースを、台帳が「公開待ち」と言い続ける状態です。

原因は、どの下書きがどのリリースなのか突き合わせる鍵が無かったことでした。  
下書きがタグを覚えていなかったため、GitHub上のリリースと照合できませんでした。

そこで、`OperationsDraft.tag` を追加しました。

リリース作成時にタグを記録し、GitHubの一次情報で台帳を書き直します。

GitHub Release側では、`draft=false` を「世に出ている」状態として扱います。

台帳は次のタイミングで直ります。

- 公開した瞬間
- `releaseInfo` のポーリング時

`releaseInfo` は30秒ごとに取り直すようにしました。  
一次情報を引けないときは、憶測で状態を倒しません。

## 見せ場

### 成功ログより一次情報を信じる

今回の中心は、成功ログを疑うことでした。

pushが成功した。  
`published: true` になった。  
公開処理が完了した。

これらは重要なログですが、読者が記事を読めることの証明ではありません。

Zenn記事なら、Zennで200が返るか。  
GitHub Releaseなら、対象タグのリリースが `draft=false` か。

外部に公開する操作では、最後に外部の一次情報を見る必要があります。

### 自律運営の神議にも、正しい世界認識が必要

神議は、台帳を見て提案します。

そのため、台帳が間違っていると、神議の提案も間違います。

今回、神議は「公開待ち4本」と言い続けていました。  
しかし、実際にはその中にZennで読める記事が含まれていました。

修正後、神議は「公開待ち4本」とは言わなくなりました。  
代わりに、`published: true` なのにZenn側で露出ゼロの記事という、本当の問題を指すようになりました。

自律的に運営する仕組みであっても、判断材料が間違っていれば正しく動けません。

### ガードは「実行前」だけでなく「実行後の確認」にも要る

AMA-terasでは、外部への発信や書き込みに承認を挟みます。

ただし、今回分かったのは、承認ゲートだけでは足りないということです。

承認して実行した後に、外部サービス側で本当に反映されたかを確認する必要があります。

今回の修正で、公開まわりのガードは次の形になりました。

1. 人間が承認する
2. アプリが実行する
3. 外部サービスの一次情報を確認する
4. 台帳を一次情報で直す
5. 神議は直った台帳をもとに判断する

「実行前に止める」だけではなく、「実行後に確認する」こともガードの一部です。

## まとめ

今回の問題は、Zennへpushしたのに読めない記事があったことから始まりました。

しかし本質は、もっと単純でした。

AMA-terasの台帳が、世界の状態ではなく、アプリ側の申告を書いていました。

その結果、

- 読めない記事を投稿済みにした
- 読める記事を公開待ちに残した
- 神議が間違った台帳を見て催促し続けた

修正では、Zennに `isLive(slug)` で確認し、200だけを `posted` にしました。  
403/404は読めないものとして扱い、不達は不明として倒さないようにしました。

さらに、過去の前提で `posted` を `staged` に戻していた `migrateStaged` を撤去しました。  
GitHub Release側にもタグを持たせ、`draft=false` を一次情報として台帳を直すようにしました。

AMA-terasの信頼は、「失敗しない」と言うことでは作れません。  
実際に起きた失敗を隠さず、原因を台帳・確認・承認のガードへ戻していく開発記録から作ります。
