「自分のコードを書き換えるAIエージェント」と聞くと、まず怖さが先に立つと思います。私も怖かったので、**壊れない仕組みから先に**作りました。

この記事は、自己進化型デスクトップエージェント **AMA-teras** を、夜間はAI(Claude Code)に無人で開発させ、朝に人間がレビューする、というサイクルを6夜回してOSS公開まで走った記録です。最終状態はテスト941件・コミット199件。夜中にAIが書いたコードでアプリが壊れたことは、一度もありませんでした。

白状すると、同じ構想は **Claude Fable 5 が出る前にも3週間やって、形になりませんでした**。今回はキャンペーンで開放されたFable 5を計画・レビュー・難所のデバッグに据えて、**一から作り直した**。すると6夜で公開まで届いた。モデルの世代が「作れるものの上限」を変える体験と、その高いモデルを破産せずに使い倒す分業の話も後半に書きます。

https://github.com/moriwo-dev-ai/ama-teras

## 何を作ったか

AMA-terasは Electron + React + TypeScript のデスクトップAIエージェントです。普通のコーディングエージェントと違うのは、**足りない機能に出会うと自分で新しいツールを生成して自分に組み込む**こと。

![デモ: 「YAMLをJSONに変換するツールが欲しい」→ 自己生成 → 検証ゲート → 承認 → その場で使える](https://github.com/moriwo-dev-ai/ama-teras/raw/main/docs/demo.gif)

デモの27秒でやっていることは実物です。「YAMLをJSONに変換したい」と頼むと、エージェントが自分にその能力が無いことを認識し、新しいツールプラグインを生成して、検証と承認を経て、その場で使えるようになります。

## なぜ壊れないか: すべての進化が同じゲートを通る

核になる設計は3つです。

**1. 稼働中の自分には触らせない(B環境)**

進化ジョブは `git worktree` で作った隔離コピー(B環境)の中だけで動きます。稼働中のアプリ(A)は無傷のまま、Bで生成→検証し、合格したものだけがAに昇格します。

```
要求 → B環境で生成 → 検証3段(typecheck → vitest → スモーク起動)
     → ユーザー承認 → 昇格(gitタグ) → 起動失敗2回で自動ロールバック
```

**2. 聖域(保護領域)**

承認機構・検証ゲート・進化エンジン自体は、進化ジョブから変更禁止です。差分が聖域に1行でも触れたら、再生成すらせず無条件拒否。「ガードを書き換えてガードを外す」という自己言及的な事故を最初に潰しておく設計です。

**3. 昇格は人間の承認制**

検証を全部通っても、最後に必ず人間のクリックが要ります。`child_process` やネットワークを使う生成ツールは、承認ダイアログで明示的に警告されます。

## 夜間無人開発の運用

開発自体もこの思想でやりました。夜、Claude Code に指示書(NIGHT_TASKS.md)を渡して寝ます。指示書には毎回、同じ規律を書きました。

- タスクごとに typecheck → テスト全緑 → コミット。動かない状態でコミットしない
- 迷ったら安全側を選び、判断理由を PROGRESS.md に記録して進む(人間は寝ているので)
- **禁止事項**: push・公開設定の変更・実設定を複製する起動

朝起きたら、PROGRESS.md の判断記録と「⚠起床後に確認してほしいこと」を読んでレビューする。この「夜はAIの自律、朝は人間の監査」というリズムが想像以上に機能しました。

## 実際に起きた事故と、防波堤が働いた話

きれいな話だけだと嘘になるので、実際にヒヤッとした話を書きます。むしろこの記事の本体はここです。

**事故1: AIが実設定を複製した**

ある夜、AIがUIの見た目確認のためにアプリを起動したのですが、方法を誤って実設定の複製(暗号化済みの鍵素材コピーを含む)を別ディレクトリに作ってしまいました。AIは自分で削除しようとしましたが、ツール権限で拒否されて残留。朝の報告で正直に申告してきたので、人間が削除しました。

対策は二段階でやりました。まず指示書の禁止事項に「実設定を複製する起動」を追加(運用での対策)。その後、検証・デモ用の一時userData起動モードを実装し、そのモードでは設定移行処理自体が走らないようにしました(**構造での対策**)。運用ルールはいつか破られるので、最後はコードで縛る。

**事故2: 旧メールアドレスがタグに残っていた**

公開前に全コミットの名義をGitHubのnoreplyアドレスに書き換えたのですが、`git filter-branch` は注釈付きタグのtagger情報を書き換えません(仕様の見落とし)。push時にGitHubのメール保護(GH007)が正しくブロックしてくれて発覚。タグを新名義で再作成して解決しました。

自前のチェックを何重にしても漏れは出ます。プラットフォーム側の防波堤(Push protection等)は全部ONにしておくべき、という教訓です。

**事故3: 自分のガードの穴を、公開直前に見つけた**

聖域トリップワイヤは進化ジョブの経路には効いていたのですが、よく見ると**通常のチャット経路からは聖域ファイルが普通の書き込み承認で通る**状態でした。自動承認+自律モードの夜間作業中なら、理論上エージェントが自分のガードを書き換えられた。

発見してすぐ、通常経路にも聖域チェックを拡張しました(自律モード中はハード拒否、手動モードは赤バナー付きの個別承認)。「ガードがある」と「ガードが全経路を覆っている」は別物でした。

**良かった話: 自分で仕込んだガードに自分が守られる**

夜間作業でAIが追加した新しいボタンのホバー色が未定義だったのを、数夜前に別件で仕込んでいたCSS回帰テストが検出して、AIが自分で直していました。ガードは積み重なると、書いた本人(AI)のミスも拾い始めます。

## コストの話: Fable 5は「ここぞ」だけ

Fable 5はAPIで入力$10/出力$50(1Mトークン)と、Opusのちょうど2倍。しかも推論ヘビーで実効コストは単価差以上に膨らみます。全部Fable 5でやる運用は(少なくとも個人には)持続しません。

開発を加速させたのは、モデルの分業です。AMA-teras自体にも6つの帯(planner / worker / explorer / reviewer / midEscalation / escalation)を実装し、計画・レビューは最上位モデル、実装の手数は中位、ファイル探索は軽量モデルに割り当てました。実測で、実行の手数の約8割を安い帯が担い、コストは約半分になりました。

ひとつ面白かったのは品質レビューの較正です。最初「4軸平均4.0以上で合格」という辛口レビューにしたら、**下位モデルの成果物が構造的に合格できず、結局ぜんぶ最上位モデルが手直しする**という本末転倒が起きました。合格基準を「重大(high)指摘ゼロ」方式に変えたら、分業が機能し始めた。強いモデルの基準をそのまま弱いモデルに当てると、分業は壊れます。

## 学び

- **承認は消すのではなく、タイミングを移す。** 夜間の無人作業では「事前の包括承認+事後の棚卸し」に置き換えると、止まらず・壊れず・説明可能になる
- **機械で検証できることにLLMを使わない。** typecheck とテストが門番の8割を担い、LLMレビューは残り2割に集中させる
- **運用ルールより構造。** 「やらないで」はいつか破られる。コードとツール権限で縛ったものだけが残る
- **AIの自律性は、ガードの堅さで買う。** ガードを増やすほど、安心して任せられる範囲が広がる

## これから

公開したのは本体だけではなく、コミュニティプラグインレジストリ(検証ゲート付き)も含みます。エージェントが生成したツールを、人間の審査と署名を通して共有し合う——「エージェント自身が書いたコードの流通」を安全にやる実験です。

正直に言うと、この開発自体がClaude Codeに支えられていて、既存のエージェントが凄いのは間違いありません。ただ、このままだと「AIエージェント=毎月固定費を払って企業から借りるもの」という形に市場が固まっていく気がしていて、そこに個人として一石を投じたいと思っています。無料枠のAPIで始められて、コードが書けない人でも「欲しい」と言えば機能が増えていく。そういうエージェントを、特定の会社ではなく、使う人みんなで育てる——AMA-terasはその実験です。
