## はじめに

AMA-terasはElectron製のデスクトップAIコーディングエージェントです。対話しながら計画→実装→デバッグ→テストを進め、足りない機能に出会うと自分でツールを生成して取り込みます。いわゆる「自己進化」です。

こう書くと本体が賢く育っていくように聞こえますが、実際に開発していて痛感するのは、進化の主役は本体ではないということです。主役は**隔離された検証**と、**最後に必ず立つ人間の承認**です。

なぜそう言い切れるのか。この記事では、開発中に実際に起きた「壊れていたのに成功と言う」系の不具合を、隠さずそのまま書きます。この製品の信頼は、賢さの主張ではなく、ガードの実話で成り立っていると考えているためです。

## 自己進化の仕組み

AMA-terasの自己進化は、大きく3段のガードを通ります。

```text
ツール生成 → git worktree(隔離) → 検証ゲート(typecheck・テスト・実起動スモーク) → 岩戸(人間の承認) → 本体へ昇格
```

### git worktreeによる隔離

エージェントが生成したツールは、本体のリポジトリから切り離した git worktree の中で動かします。生成物がビルドを壊しても、テストを落としても、本体には波及しません。壊れることを前提に、壊れてよい場所を用意しています。

### 検証ゲート(typecheck・テスト・実起動スモーク)

worktree内の生成物は、typecheck・テスト・実起動スモークの3つの検証を通らないと昇格候補になりません。ここで実際に起きた失敗を2つ書きます。

**M51: 進化ゲートを決定的にする**

昇格判定が落ちる事象を調べたところ、原因は生成物ではなく、**生成物と無関係なテストのフレーク**でした。フレーク(並列負荷などで不安定になるテスト)はこのプロジェクトでも実在します。たとえば別マイルストーンのテスト実行でも「全1160緑(実プロセス系2件は並列負荷のフレーク、単独で緑)」という記録が残っています。無関係な不安定要素で進化の可否が変わるのは判定として成立しないため、ゲートを決定的(deterministic)にする修正を入れました。

**M50: 進化ジョブが「作られる前に」死ぬ穴**

進化ジョブが、実行以前——ジョブが作られる段階で死ぬケースがありました。原因は残骸環境とIDの再利用の組み合わせです。過去の実行の残骸が残った環境で同じIDが再利用されると、新しいジョブが成立しません。この穴を塞ぎました。

どちらも「生成されたコードが悪い」失敗ではなく、**ゲート自身の信頼性**の失敗です。ゲートが信用できなければ、その先の承認にも意味がなくなります。

### 岩戸(承認ゲート)

検証を通った生成物でも、本体への昇格は必ず人間の承認ダイアログを通します。外部への発信・書き込みも同様で、承認なしには実行できない設計です。この承認ゲートを「岩戸」と呼んでいます。

この思想はコーディング以外の機能にも一貫させています。たとえば音声から約束・ToDoを抽出する機能(M42)でも、抽出結果を記録に勝手に書くことはせず、確認UIで人が承認したものだけが入ります。

ここでも実話があります。

**M66: 「実行できない提案」を承認済み扱いで流していた**

そもそも実行できない提案が、承認フローに乗り、承認済み扱いで流れていました。承認したのに何も起きない——これは承認の意味を空洞化させます。**承認は実行できるものだけを対象にする**よう修正しました。承認ゲートは「置けばよい」ものではなく、承認対象の実行可能性まで含めて初めて機能します。

## 運営の自律(神議)とその失敗談

AMA-terasには「神議」と呼ぶ、プロジェクト運営(SNS運用・週報・トリアージなど)を担う自律の仕組みがあります。そしてここが、タイトルの「壊れていたのに成功と言う」問題が最も濃く出た場所です。

**M58: 働いていない神が、働いているように見えていた**

Bluesky検索が403を返していたのに、そのエラーが握りつぶされていました。結果として、検索担当は何も検索できていないのに、外からは正常に働いているように見えていました。エラーを飲み込んで沈黙するコードは、「失敗」より悪い「偽の成功」を作ります。

**M61 / M63 / M64 / M65: 誤読・言行不一致・状態誤認の連鎖**

M58の修正後も、誤読・言行不一致・状態誤認といった不具合が連鎖的に見つかりました。1つ直すと、それに隠れていた次の不整合が露出する、という順で発見が続きました。

ここから得た方針はシンプルです。「壊れているのに動いていると報告する」問題は、AIエージェントである以上ゼロにはできません。だからAMA-terasは、これを**隠す設計ではなく、検出して直す設計**に寄せています。エラーを握りつぶさない、報告と実態の突き合わせが可能な形にする、見つかった不具合はマイルストーンとして記録する——この繰り返しです。

余談ですが、「無いものを有ると報告する」現象はLLMの応答に限りません。ローカル文字起こし(whisper.cpp v1.9.1 + ggml-small)の実機検証では、2秒の無音に対して「(音楽)」という幻聴が返ってきました。そのまま通すと記録がゴミで埋まるため、フィルタを追加しています。どの層にも「偽の成功」の芽はあり、層ごとに検出を置くしかありません。

## 未公開情報を漏らさないガード

自律的に外へ発信する仕組みを持つ以上、未公開情報の漏洩は構造的なリスクです。ここは多重に守っています。

- **M59: 三重チェック** — ①発信の素材から未公開情報を抜く、②プロンプトで言及を禁じる、③出力段階で検出して捨てる。1層が抜けても残り2層で止まる構成です。
- **M49: 公開前チェックリスト** — 未公開機能が公開経路に紛れ込まないよう、公開前のチェックリストを運用しています。
